Case Study

bug-triage-ai

Turning unstructured bug reports into severity-scored, Jira-ready tickets in seconds — with a confidence gate that keeps a human in the loop on anything uncertain.

github.com/samyamdhamala/bug-triage-ai / no CI
01

The problem

Bug intake is manual, inconsistent, and slow. Reports arrive scattered across Slack, email, and raw notes, each written differently, and someone has to read, interpret, and rewrite every one of them into a structured ticket before an engineer can act on it. That translation step is where triage time actually goes.

This MVP automates that translation: raw text in, a structured Jira-style ticket out, in seconds instead of minutes — while keeping a human explicitly in the loop on anything the model isn't confident about.

02

Architecture

A three-tier pipeline: an intake surface (Streamlit UI or a Slack /triage command), a FastAPI backend that owns the LLM abstraction and triage logic, and a deterministic post-processing layer that applies rules an LLM shouldn't be trusted to get consistently right on its own.

[ Streamlit UI ] or [ Slack #bugs · /triage ] │ ▼ FastAPI backend ──► LLM layer (provider-agnostic) │ OpenAI · Groq · Gemini ▼ Post-LLM deterministic rules team routing · label injection │ ▼ Confidence gate High → auto-file ticket Medium → quick human scan Low → blocked, routed to a human │ ▼ [ JSON export ] + [ Jira ticket, with duplicate pre-check ]

The repo is split cleanly along this pipeline: backend/ for the FastAPI service and LLM abstraction, frontend/ for the Streamlit demo UI, prompt.txt holding the exact LLM instructions as a reviewable artifact rather than a buried string, and outputs/ for the serialized JSON results.

03

Key decisions & tradeoffs

  • Provider-agnostic LLM layer

    Supports OpenAI (GPT-4o-mini), Groq (Llama 3.1-70B), and Gemini behind one interface, configured via .env (LLM_PROVIDER + API key). Costs a bit of abstraction overhead but means swapping providers for cost or latency reasons never touches the frontend or backend logic.

  • Deterministic rules after the LLM, not inside it

    Team routing and label injection happen in plain code after the model responds, instead of being folded into the prompt. It's more code to maintain when org structure changes, but it removes an entire class of hallucination risk from a decision that has to be consistent every time.

  • Three-tier confidence gate instead of pass/fail

    Every ticket gets scored High, Medium, or Low rather than a binary accept/reject. Medium gets a quick human scan; Low is blocked from auto-creation entirely. It trades full automation for something that fails safe.

04

Confidence scoring & severity rubric

Severity and confidence are both surfaced explicitly on every generated ticket, so a reviewer can tell at a glance what needs attention. A well-structured report scores High; a vague one automatically scores Low. The one hard rule that overrides the gate: P1-severity tickets always get reviewed, regardless of confidence.

Title: Login hangs on spinner after credential entry Severity: P2 Confidence: High Component: Auth
05

Duplicate detection

Before filing, the system searches open Jira tickets by label to catch obvious repeats. It's a deliberately shallow first pass — label matching rather than semantic similarity — which is an acknowledged limitation, not an oversight. Vector-similarity matching is the next planned iteration once the label-based version proves out the rest of the pipeline.

06

Where it can fail

Documented openly in the README rather than glossed over — the honesty here was intentional, since a triage tool that hides its own failure modes is worse than a manual process.

  • Guessed repro stepsGenerated reproduction steps are inferred from the report text and need human validation before they're trusted.
  • Multi-bug reportsA single report describing several distinct bugs isn't split automatically — it's treated as one ticket.
  • Retry logic is syntactic, not semanticAuto-retry catches JSON parsing failures, not cases where the model's reasoning itself is wrong.
07

Takeaway

Operational velocity and explainability, not perfect automation — composing LLM extraction with deterministic rules and a confidence gate to capture real intake gains without spamming the tracker with false positives.
