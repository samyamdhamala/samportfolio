Case Study

bug-triage-ai

Turning unstructured bug reports — and now raw CI test failures — into severity-scored, Jira-ready tickets in seconds, with semantic duplicate detection and a confidence gate that keeps a human in the loop on anything uncertain.

github.com/samyamdhamala/bug-triage-ai /
01

The problem

Bug intake is manual, inconsistent, and slow. Reports arrive scattered across Slack, email, and raw notes — and just as often as a wall of failing CI output — each in a different shape, and someone has to read, interpret, and rewrite every one of them into a structured ticket before an engineer can act on it. That translation step is where triage time actually goes.

This system automates that translation across every intake path: raw text or a JUnit XML failure report in, a structured, deduplicated, severity-scored ticket out, in seconds instead of minutes — while keeping a human explicitly in the loop on anything the model isn't confident about.

02

Architecture

Three intake surfaces feed one pipeline: a Streamlit UI, a Slack /triage command, and — new in v2 — a POST /triage/ci endpoint that ingests JUnit XML straight from a CI run. From there, a FastAPI backend owns the LLM abstraction and triage logic, followed by a deterministic rules layer and a two-layer duplicate check before anything reaches the confidence gate.

[ Streamlit UI ] · [ Slack /triage ] · [ CI: POST /triage/ci (JUnit XML) ] │ ▼ FastAPI backend ──► LLM layer (provider-agnostic, structured JSON output) │ OpenAI · Groq · Gemini · Ollama ▼ Deterministic rules engine team routing · severity escalation · test-area suggestions │ ▼ Two-layer duplicate check local embeddings (cosine ≥0.68) → Jira semantic search (≥0.72) │ ▼ Confidence gate High → auto-file ticket Medium → quick human scan Low → blocked, routed to a human │ ▼ [ JSON export ] + [ Jira ticket ] + [ /feedback correction replayed into future prompts ]

The repo is organized around this pipeline: backend/triage.py orchestrates prompt → LLM → parse → rules, backend/llm_client.py is the provider-agnostic LLM wrapper, backend/rules.py holds the deterministic routing/severity logic, backend/vector_store.py and backend/jira_client.py implement the two dedup layers, backend/ci_parser.py handles JUnit XML ingestion, and backend/feedback_store.py stores QA corrections that get replayed into later prompts.

03

Key decisions & tradeoffs

  • CI failures as a first-class input, not just human reports

    The /triage/ci endpoint accepts JUnit XML plus branch/commit/run-URL metadata directly from a pipeline, so a red build gets the same structured triage as a Slack message. It turns the tool from "bug report formatter" into a triage layer that sits in front of both humans and CI.

  • Two-layer semantic dedup over label matching

    v1 matched duplicates by label only — a known, deliberate shortcut. v2 replaces it with local sentence-transformer embeddings (cosine ≥0.68) as a fast first pass, then a Jira semantic search (≥0.72) before filing. More infrastructure to run, but it catches paraphrased repeats that label matching structurally can't.

  • Provider-agnostic LLM layer with enforced structured output

    OpenAI, Groq, Gemini, and Ollama sit behind one interface in llm_client.py, swappable via .env, each forced into the same JSON schema regardless of the provider's native format. Costs abstraction overhead but means a provider swap for cost, latency, or even fully local inference never touches the rest of the pipeline.

  • Deterministic rules after the LLM, not inside it

    Team routing, severity escalation, and test-area suggestions happen in plain code after the model responds, instead of being folded into the prompt. More code to maintain when org structure changes, but it removes an entire class of hallucination risk from decisions that have to be consistent every time.

  • QA feedback loop instead of retraining

    A /feedback BUG-42 <correction> command stores QA corrections and replays them into future prompts. It's a cheap approximation of "learning" — no fine-tuning pipeline required — that lets the system absorb human judgment over time.

  • Three-tier confidence gate instead of pass/fail

    Every ticket gets scored High, Medium, or Low rather than a binary accept/reject. Medium gets a quick human scan; Low is blocked from auto-creation entirely. It trades full automation for something that fails safe.

04

Confidence scoring & severity rubric

Severity (P1–P4) and confidence are both surfaced explicitly on every generated ticket, so a reviewer can tell at a glance what needs attention. Deterministic rules bump severity when a report contains phrases like "all users" or "production down," and generated reproduction steps are prefixed [generated] as a signal they still need human validation. The one hard rule that overrides the gate: P1-severity tickets always get the urgent-review label, regardless of confidence.

05

Duplicate detection

v1 only matched duplicates by label — an acknowledged shortcut, not a final answer. v2 replaces it with a two-layer semantic check: a local vector store (sentence-transformers) does a fast first pass at cosine similarity ≥0.68, then a Jira semantic search confirms against open tickets at ≥0.72 before anything gets filed. A match short-circuits ticket creation and surfaces the similarity score instead, so paraphrased repeats — "login times out" vs. "login hangs on spinner" — get caught instead of filed twice.

06

Where it can fail

Documented openly in the README rather than glossed over — the honesty here was intentional, since a triage tool that hides its own failure modes is worse than a manual process. Structured-output enforcement in v2 cut down JSON parsing failures significantly, but the remaining gaps are real:

  • Guessed repro stepsGenerated reproduction steps are inferred from the report text and prefixed [generated], but still need human validation before they're trusted.
  • Multi-bug reports get flattenedA report describing several distinct bugs now gets a note that it "contains multiple issues," but still triages to a single ticket at the most severe level rather than splitting automatically.
  • No calibration or metrics yetThere's no dashboard tracking routing accuracy, confidence calibration, or bugs/day — the roadmap has it, but today the only feedback loop is the manual /feedback command.
  • Storage is still flat JSONFeedback, embeddings, and outputs all persist to local files, not Postgres. Fine at demo scale, a real constraint before this could run unattended in production.
07

Takeaway

From MVP to a real triage layer: the same bet on operational velocity and explainability over full automation, now extended to meet bugs where they actually originate — a Slack message, a manual report, or a red CI pipeline — without losing the human-in-the-loop guarantees that made v1 trustworthy.
